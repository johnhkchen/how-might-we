## Context

ExpandHMW generates 6-8 variants per call and RefineHMW generates a similar number. With concise rationales (from T-008-02), 4 variants per call is enough — users can always "Go Deeper" for more. This cuts token usage per expansion/refinement call by ~40%.

## Changes Required

### BAML prompts
- `expand.baml` — change "Generate 6-8 variants" to "Generate exactly 4 variants with diverse moves"
- `refine.baml` — change generation instructions to "Generate 3-4 new variants"

### Frontend
- Update any UI text or skeleton placeholders that assume 6+ variants
- The VariantGrid skeleton shows 3 placeholder cards — keep as-is (matches 4 variants)

### Mock fixtures
- Reduce mock expansion partials to 4 variants
- Reduce mock refinement partials to 3-4 variants

## Acceptance Criteria

- [ ] ExpandHMW produces exactly 4 variants per call
- [ ] RefineHMW produces 3-4 new variants per call
- [ ] Each variant still has a distinct `moveType` (no duplicate move types within a single expansion)
- [ ] "Go Deeper" still works for additional iterations
- [ ] Mock fixtures updated to 4 variants
- [ ] Playwright tests pass with updated fixture counts
