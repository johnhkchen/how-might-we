## Context

Finalize the SST configuration for deploying the Go backend as a Lambda function with streaming Function URL. The `sst.config.ts` is scaffolded — this ticket makes it actually deploy. Includes adding the Lambda streaming adapter to the Go code and setting up secrets via SSM.

**Known blocker**: BAML's Go bindings require CGo (`//go:build cgo`), which doesn't cross-compile from macOS to linux. SST's Go runtime does `GOOS=linux GOARCH=amd64` without CGo. Solutions to evaluate:
1. Build the binary in a Docker container and deploy as a container Lambda (Dockerfile in `backend/`)
2. Use `sst.aws.Function` with a custom build command that invokes Docker
3. Use a CI/CD pipeline (GitHub Actions with Linux runner) for deployment
4. Investigate if BAML has a pure-Go mode without CGo

AWS credentials are configured via `credential_process` bridge in `~/.aws/config` (profile `sst`). SST secret `AnthropicApiKey` is already set for stage `dev`.

## Acceptance Criteria

- [ ] Go backend includes Lambda streaming adapter (wraps the same `http.ServeMux`)
- [ ] CGo cross-compilation solved (Docker build, CI pipeline, or container Lambda)
- [ ] `npx sst deploy` successfully creates the Lambda function with streaming enabled
- [ ] Function URL is accessible and returns SSE responses
- [ ] Lambda reads API key from SST-linked secret (AnthropicApiKey already set)
- [ ] Cold start is under 500ms (relaxed from 200ms due to container/CGo overhead)
