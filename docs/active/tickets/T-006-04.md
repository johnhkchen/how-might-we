## Context

The CF Worker rate limiting is per-isolate (in-memory maps). An attacker hitting different edge locations or triggering new isolate spawns can bypass limits. For real protection, rate limit state needs to be shared — either via Cloudflare KV (eventual consistency, cheap) or Durable Objects (strong consistency, more expensive).

KV is sufficient for this use case — rate limits don't need to be perfectly precise.

## Acceptance Criteria

- [ ] Rate limit counters stored in Cloudflare KV namespace
- [ ] Per-IP limit (20 req/min) enforced globally, not per-isolate
- [ ] Per-session limit (50 calls) enforced globally
- [ ] KV namespace created and bound in `wrangler.toml`
- [ ] Rate limit headers (`X-RateLimit-Remaining`, `X-RateLimit-Reset`) still present
- [ ] Fallback to in-memory if KV is unavailable (graceful degradation)
- [ ] KV costs documented (free tier: 100k reads/day, 1k writes/day)
