## Context

The analysis stage produces a single `initialReframing` string. Users naturally use it as their next HMW, but then the expansion critiques it ("too narrow", "too broad") — a bad "you told me to pick it!" experience.

Fix: replace the single reframing with 3 short options at different scope levels (narrower, similar, broader). The user picks one, and that feeds into expansion. This makes the transition from analysis → expansion feel like a deliberate choice, not a trap.

## Changes Required

### BAML types (`types.baml`)
Replace `initialReframing string` in `HMWAnalysis` with:
```
class ReframeOption {
  statement string @description("the reframed HMW question, one sentence")
  scope "narrower" | "similar" | "broader"
  whyThisScope string @description("one line: why this scope level opens different solutions")
}
```
And change `HMWAnalysis.initialReframing` to `reframeOptions ReframeOption[]`.

### BAML prompt (`analyze.baml`)
Update prompt to generate exactly 3 reframe options — one narrower, one at similar scope, one broader. Each should be a single sentence HMW question with a one-line rationale.

### Frontend (`AnalysisPanel.svelte`)
Replace the single reframing display with 3 selectable cards. Clicking one highlights it and stores the selection.

### Frontend (`workshop/+page.svelte`)
The selected reframe option feeds into `expandHMW()` as the basis for expansion (replacing the old `initialReframing`).

### Go types
Run `baml-cli generate` to update the Go client. The handler doesn't need changes since it just streams the BAML output.

## Acceptance Criteria

- [ ] Analysis returns 3 reframe options (narrower, similar, broader)
- [ ] Each option is a single sentence + one-line scope rationale
- [ ] User can click to select one of the 3 options
- [ ] Selected option is visually highlighted
- [ ] Expansion uses the selected reframe as its starting point
- [ ] If no option is selected, the "Expand" button is disabled
- [ ] Mock fixtures updated with 3-option format
- [ ] Playwright tests updated
