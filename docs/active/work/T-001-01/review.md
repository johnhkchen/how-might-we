## Deviations from Original Ticket

The ticket assumed BAML sources were ready for generation. Two issues required fixes:

1. **`move` reserved keyword**: The `HMWVariant.move` field used a BAML language keyword. Renamed to `moveType`. This is a schema change that affects:
   - Frontend code that consumes HMWVariant JSON (field name changes from `move` to `moveType`)
   - Any BAML prompt tests referencing the field
   - The specification document (still references `move`)

2. **Jinja template syntax**: The `??` null-coalescing operator is not valid Jinja2. Replaced with standard `if/else` conditional expressions.

3. **Version bump**: BAML CLI v0.218.0 was installed but project specified v0.216.0. Aligned both generators.baml and go.mod to v0.218.0.

---

## Open Concerns

1. **`moveType` rename propagation**: The field rename from `move` to `moveType` must be reflected in:
   - `docs/specification.md` — still references `move`
   - Frontend type definitions (when they're built)
   - Any other tickets that reference `HMWVariant.move`

   This is a low-risk concern since the frontend isn't built yet, but downstream tickets should use `moveType`.

2. **baml_client/ is gitignored**: The generated client must be regenerated by any developer or CI pipeline before building. The `baml-cli generate` command is documented in CLAUDE.md. CI configuration should include this step.

3. **goimports PATH issue**: The `on_generate` hook in generators.baml expects `goimports` on PATH. It's installed at `~/go/bin/goimports` which may not be on PATH in all shells. The hook failure is non-blocking (files are still generated), but it means `go mod tidy` may need to run manually after generation.

4. **Go version floor raised**: `go mod tidy` updated the minimum Go version from 1.23 to 1.24.0. This is driven by the BAML SDK v0.218.0's requirements. All developers need Go 1.24+.
